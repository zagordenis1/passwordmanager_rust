# Code audit — 2026-04

Full-program audit done after merging PRs #1 and #2. The four bugs
addressed in PR #2 are not repeated here. Below are the **new** findings,
graded H/M/L by impact. Items marked **Fixed in PR #3** are addressed in
this same PR; the rest are documented for future work.

---

## H1. Race in `set_master_password` between concurrent processes
**Status: Fixed in PR #3 (`src/manager.rs`).**

`has_master_password()` and the three meta upserts ran in two separate
SQLite operations. Two `pwm init` processes could both see "no master
yet" and then race on writing salt / verifier / kdf_version. The
`INSERT … ON CONFLICT DO UPDATE` pattern keeps each row internally
consistent, but it does **not** prevent one process from writing
`salt` from its derivation while the other writes `verifier` from
its — leaving the DB unrecoverable (the salt can no longer decrypt the
verifier).

**Fix.** Wrap the writes in a `BEGIN IMMEDIATE` transaction (`rusqlite`'s
`TransactionBehavior::Immediate`) and re-check `salt` + `verifier`
presence inside the transaction. The first process to acquire the
reserved lock writes; the second sees the master is set and bails.

Regression test: `manager::tests::second_init_in_face_of_already_set_master_bails`.

## H2. Exported JSON gets default 0644 permissions
**Status: Fixed in PR #3 (`src/manager.rs::write_owner_only`).**

`std::fs::write` honours the user's umask. On a typical desktop
(`umask 022`) the export file lands at mode `0644` — world-readable.
The file contains every plaintext password.

**Fix.** New `write_owner_only` helper opens the file with
`OpenOptionsExt::mode(0o600)` on Unix; on Windows it falls through to
`std::fs::write` (existing-file ACLs are preserved by overwrite, and
new files inherit directory ACLs). Both CLI paths (`pwm export` and
the interactive menu's item 6) now also print a stderr/stdout warning
that the file is plaintext and should be deleted after use.

Regression test: `manager::tests::export_file_has_owner_only_mode`
(Unix-only; gated on `#[cfg(unix)]`).

## H3. `import_from_json` silently stored empty passwords
**Status: Fixed in PR #3 (`src/manager.rs::import_from_json`).**

Previous: any entry with a present-but-empty `password` field was
encrypted and stored. The Fernet token of the empty string is fine, but
the resulting account is functionally junk and will silently appear in
`list` / `search` output.

**Fix.** The malformed-entry filter now also rejects empty `password`.
Existing handling of missing `login` is preserved.

Regression test: `manager::tests::import_skips_entries_with_empty_password`.

---

## M1. `self.key` field on `PasswordManager` was dead and misleading
**Status: Fixed in PR #3 (struct change in `src/manager.rs`).**

The struct used to hold both `Option<Fernet>` and a parallel
`Option<SecretString>` of the base64 key. The `SecretString` was
written but never read — its only effect was zeroising the base64 key
copy on Drop.

The misleading part: it implied defense-in-depth, but `fernet::Fernet`
itself stores the AES-128 key + HMAC-SHA256 key in plain (non-zeroising)
buffers, so as long as the manager is unlocked the actual key material
already lives unprotected in heap memory regardless of `SecretString`.

**Fix.** Drop the field. Explicitly `zeroize` local base64 key strings
in `set_master_password` / `verify_master_password` /
`change_master_password` after the `Fernet` is built. Add a doc note on
the struct explaining the upstream-crate limitation. Drop the now-unused
`secrecy` dependency from `Cargo.toml`.

## M2. No SQLite durability PRAGMAs
**Status: Fixed in PR #3 (`src/db.rs::open`).**

The default journal mode (`DELETE`) has a known mid-rename corruption
window on power loss; the default `synchronous` value depends on the
build (`FULL` on most, but not guaranteed). For a password store the
correct policy is durability-first.

**Fix.** `db::open` now sets:
- `journal_mode=WAL` — corruption-safe under crash; readers and writers
  can coexist (cheap insurance).
- `synchronous=FULL` — explicit fsync on every commit, regardless of
  build defaults.
- `foreign_keys=ON` — defensive default for future schema changes.

No regression test (PRAGMA values are environment-state; covered by the
existing CRUD tests still passing under WAL).

## M3. Auto-lock cannot preempt a blocking menu read
**Status: Documented only — non-trivial fix.**

`ensure_unlocked` is checked **after** the user has already typed a
menu choice. If the binary is left at the "Виберіть пункт:" prompt,
`stdin.read_line` blocks indefinitely and the master key remains in
RAM no matter how long the user is away from the keyboard. The
`PM_AUTO_LOCK_SECONDS` semantics today only cover *between-action*
idleness, not *during-prompt* idleness.

A real fix would either:
1. Use a non-blocking read with a timeout (`mio` / `crossterm` / a
   poll-and-retry loop on Unix; `WaitForSingleObject(GetStdHandle)` on
   Windows), or
2. Spawn a watchdog thread that calls `manager.lock()` after the
   timeout, then prints "lock — re-auth" once the user finally types
   something.

Both deviate from the Python reference's UX and add platform-specific
machinery; deferred until the user explicitly requests it.

## L1. No master-password length / complexity gate
**Status: Documented only — matches Python reference.**

`set_master_password` and `change_master_password` accept any non-empty
string. `"a"` is acceptable. Adding a minimum length / class gate would
deviate from the Python reference (which has the same behaviour) and
could break user-scripted scenarios; leaving alone.

---

## L2. `prompt_line` strips leading whitespace
**Status: Documented only.**

`src/cli/interactive.rs::prompt_line` calls both `trim_end_matches`
and `.trim()`. A login prefixed with whitespace is silently de-prefixed.
Cosmetic; leaving alone (passwords use `prompt_password` which does
NOT trim).

---

## L3. Generator's `MIN_LENGTH=4` is very low
**Status: Documented only — matches Python reference.**

`pwm gen --length 4` works. With four required classes and length 4,
the generator emits exactly one of each. `4` is below NIST SP 800-63B
recommendations for memorised secrets, but `pwm` is a *manager* — the
user sets their own policy. Worth a README note in a future doc PR.

---

## L4. Verifier UTF-8 round-trip uses `from_utf8`, not byte compare
**Status: Documented only — correct as-is.**

`verify_master_password` reads the verifier blob, decodes to UTF-8, and
passes it to `Fernet::decrypt`. Any non-UTF-8 verifier triggers an
error path that bails out. Since Fernet tokens are always ASCII base64
this branch is unreachable on a non-corrupt DB. Documented, no change.
