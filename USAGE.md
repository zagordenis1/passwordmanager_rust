## Швидкий старт за 60 секунд

```bash
# 1) Збірка (один раз)
cd passwordmanager_rust
cargo build --release

# 2) Виставити master-пароль (перший раз)
./target/release/pwm init
#   → введіть новий master-пароль двічі

# 3) Додати акаунт
./target/release/pwm add --login github --email me@example.com
#   → введіть master-пароль, потім сам пароль для github

# 4) Отримати пароль
./target/release/pwm get github
#   → введіть master-пароль; пароль виводиться у stdout

# 5) Або запустіть інтерактивне меню (не вказуючи підкоманду)
./target/release/pwm
```

Якщо встановили глобально (`cargo install --path .`), команди
скорочуються до `pwm init`, `pwm add ...` тощо.

## Концепції, які треба розуміти

| Поняття              | Що це                                                                                                  |
|----------------------|--------------------------------------------------------------------------------------------------------|
| **Master-пароль**    | Один-єдиний пароль, який ви запам'ятовуєте. Усе інше шифрується похідним від нього ключем.              |
| **БД (`users.db`)**  | SQLite-файл, що містить зашифровані записи. Без master-пароля — нечитаємий.                             |
| **Verifier**         | Зашифрований відомий plaintext, що дозволяє перевірити правильність master-пароля без розшифровки даних. |
| **KDF**              | Argon2id (за замовчуванням) або PBKDF2 (legacy). Перетворює master-пароль на ключ шифрування.            |
| **Fernet**           | Формат симетричного шифрування (AES-128-CBC + HMAC-SHA256), сумісний з Python `cryptography.fernet`.    |
| **Auto-lock**        | Автоматичне скидання ключа з пам'яті після N секунд бездіяльності в інтерактивному меню.                |
| **Запис (record)**   | Один акаунт: `id`, `login`, `email`, `password`, `created_at`. Тільки `password` зашифровано.            |

## Розташування БД

За замовчуванням БД створюється у файлі `users.db` у **поточній теці**
(там, звідки ви запускаєте `pwm`). Це **не** глобальне сховище в
`~/.config` чи десь іще.

Шлях до БД задається прапорцем `--db`, який можна вказати з будь-якою
підкомандою (включно з відсутньою — тоді запускається меню):

```bash
# Кожна команда — на іншу БД
pwm --db ./personal.db init
pwm --db ./work.db init
pwm --db /home/me/.local/share/pwm/secrets.db get github

# Сумісно з інтерактивним меню
pwm --db ~/passwords/main.db
```

> **Порада.** Якщо ви хочете "глобальне" сховище — створіть alias у
> shell:
> ```bash
> alias pwm='command pwm --db "$HOME/.local/share/pwm/users.db"'
> ```
> і додайте його в `~/.bashrc` / `~/.zshrc`.

При першому використанні `--db <шлях>`:
- Якщо файл не існує — він буде створений (файл + sidecar-файли SQLite WAL).
- Якщо тека не існує — `pwm` поверне помилку. Створіть її заздалегідь.

WAL-режим створює два додаткових файли поряд з `users.db`:
`users.db-wal` і `users.db-shm`. Це нормально — SQLite сам ними
керує. При закритті БД дані WAL зливаються у головний файл.

## Інтерактивне меню — пункт за пунктом

Запуск без підкоманди:

```bash
pwm                           # users.db у поточній теці
pwm --db ~/secrets/main.db    # власний шлях
```

Якщо master-пароль ще не задано — пункт меню одразу запропонує його
створити. Після вводу master-пароля відкривається меню:

```
=== Password Manager ===
1)  Додати акаунт
2)  Знайти акаунт
3)  Показати всі акаунти
4)  Видалити акаунт
5)  Оновити пароль
6)  Експорт у JSON
7)  Імпорт з JSON
8)  Пошук по login/email
9)  Вихід
10) Змінити master password
11) Згенерувати пароль
Виберіть пункт:
```

### 1 — Додати акаунт

Запитує `Login`, `Email`, `Password`. У полі пароля можна ввести
**`g`** замість справжнього пароля — `pwm` запитає параметри (довжина,
класи символів) і згенерує. Згенерований пароль буде відображено та
збережено.

```
Login: github
Email: me@example.com
Password (або 'g' щоб згенерувати): g
Довжина [20]:
Нижній регістр (a-z)? [Y/n]:
Верхній регістр (A-Z)? [Y/n]:
Цифри (0-9)? [Y/n]:
Символи (!@#…)? [Y/n]:
Згенеровано: KQ9pRm-3xZjL!t8uYwBe
Створено:
  id=4 login="github" email="me@example.com" password="KQ9pRm-3xZjL!t8uYwBe" created_at=...
```

### 2 — Знайти акаунт

Точне співпадіння по `login`. Виводить весь запис, включно з
розшифрованим паролем.

### 3 — Показати всі акаунти

Перелік усіх записів. Якщо БД порожня — `(порожньо)`.

### 4 — Видалити акаунт

Запитує login + підтвердження `[y/N]`. Будь-яка відповідь крім
`y`/`yes`/`т`/`так` — скасування.

### 5 — Оновити пароль

Запитує login + новий пароль (можна `g` для генерації, як у пункті 1).

### 6 — Експорт у JSON

Запитує шлях файлу (за замовчуванням `export.json` у поточній теці).
Записує всі акаунти у вигляді JSON-масиву з розшифрованими паролями.
**На Unix** файл створюється з правами `0600` (тільки власник). Виводиться попередження про відкриті паролі.

```
Виберіть пункт: 6
Шлях до файлу експорту [export.json]: ~/backup/secrets.json
Експортовано 12 акаунтів у ~/backup/secrets.json.
Увага: файл містить ВСІ паролі у відкритому вигляді. На Unix його
створено з режимом 0600 (тільки власник). Видаліть його після
використання.
```

### 7 — Імпорт з JSON

Запитує шлях файлу. Читає JSON-масив, шифрує кожен запис **поточним**
master-паролем (а не master-паролем БД-джерела), вставляє в БД.
Дублікати по `login` пропускаються; інші помилки повертаються одразу.

```
Виберіть пункт: 7
Шлях до файлу імпорту: ~/backup/secrets.json
Імпортовано 12 нових акаунтів (дублікати пропущено).
```

### 8 — Пошук по login/email

Підрядковий пошук case-insensitive (для ASCII). `%` та `_` екрануються
автоматично, тож пошук по `%` повертає лише записи з literal `%`.

### 9 — Вихід

Друкує "До побачення." і завершується з кодом 0. **Не вимагає**
повторної автентифікації, навіть якщо auto-lock вже спрацював.

### 10 — Змінити master password

Запитує:
1. Поточний master-пароль (для перевірки).
2. Новий master-пароль.
3. Підтвердження нового master-пароля.

Усі рядки в БД **атомарно перешифровуються** новим ключем у межах
однієї транзакції. Якщо щось зламається посередині — БД лишиться у
старому консистентному стані.

> Це **єдиний** шлях мігрувати legacy-БД (PBKDF2) на сучасний Argon2id.

### 11 — Згенерувати пароль

Як пункт 1, але без створення запису в БД — просто друкує згенерований
пароль. **Не скидає** auto-lock-таймер і **не вимагає** master-пароля.

## Не-інтерактивні підкоманди — повний довідник

Усі підкоманди підтримують глобальний прапорець `--db <шлях>`.
Master-пароль читається з:
- **прихованого prompt'а** (без луни) — якщо `stdin` це TTY;
- **одного рядка stdin** — якщо `stdin` це pipe / файл (зручно для скриптів).

### `pwm init` — перше налаштування

```bash
pwm init                          # запитає master + підтвердження
echo -e "master\nmaster" | pwm init   # неінтерактивно, для CI
```

Падає з помилкою, якщо master-пароль вже встановлено.

### `pwm add --login <name> [--email <e>] [--stdin]`

```bash
pwm add --login github --email me@example.com
# → master-prompt → password-prompt

# Pipe master + password по одному рядку:
printf 'master\nmysecretpw\n' | pwm add --login github --email me@example.com --stdin
```

Без `--stdin` — пароль читається з прихованого prompt'а (TTY) або з
другого рядка stdin (не-TTY). З `--stdin` — пароль обов'язково з
другого рядка stdin (без prompt'а), що зручно для CI.

### `pwm get <login> [--full]`

```bash
pwm get github                # тільки пароль у stdout
pwm get github --full         # повний запис у stdout
echo "master" | pwm get github | pbcopy     # macOS clipboard
echo "master" | pwm get github | wl-copy    # Wayland clipboard
echo "master" | pwm get github | xclip -selection clipboard   # X11
```

Якщо логін не знайдено — друкує `Не знайдено.` у stderr і виходить з
кодом `1`.

### `pwm list [--json]`

```bash
pwm list                       # людино-читаний вивід
pwm list --json | jq '.'       # JSON, обробити jq
pwm list --json | jq '.[] | .login'   # тільки логіни
```

### `pwm rm <login> [--force]`

```bash
pwm rm github                   # питає підтвердження y/N
pwm rm github --force           # без підтвердження (для скриптів)
```

### `pwm update <login> [--stdin]`

```bash
pwm update github
# → master-prompt → новий password-prompt

printf 'master\nnew-secret\n' | pwm update github --stdin
```

### `pwm change-master`

```bash
pwm change-master
# → старий → новий → підтвердження нового

# Pipe для CI (3 рядки):
printf 'old\nnew\nnew\n' | pwm change-master
```

Атомарна операція: усі рядки перешифровуються в одній транзакції.

### `pwm gen [--length N] [--no-lower] [--no-upper] [--no-digits] [--no-symbols]`

```bash
pwm gen                       # 20 символів, усі класи
pwm gen --length 32           # 32 символи
pwm gen --length 24 --no-symbols    # літери + цифри
pwm gen --length 16 --no-symbols --no-upper   # тільки нижній регістр + цифри
```

Не торкається БД, не запитує master-пароль.

### `pwm search <query> [--json]`

```bash
pwm search github             # підрядковий пошук
pwm search "@example.com" --json | jq
```

### `pwm export <path>`

```bash
pwm export backup.json
pwm export ~/backups/$(date +%Y%m%d)-pwm.json
```

Деталі — нижче в розділі ["Експорт / імпорт у JSON"](#експорт--імпорт-у-json--деталі).

### `pwm import <path> [--no-skip-duplicates]`

```bash
pwm import backup.json                          # дублікати пропускаються
pwm import backup.json --no-skip-duplicates     # перший дублікат → помилка
```

### Прапорці верхнього рівня

```bash
pwm --version           # версія
pwm --help              # перелік підкоманд
pwm <команда> --help    # довідка по конкретній підкоманді
```

## Експорт / імпорт у JSON — деталі

Я перевірив наступні сценарії наживо на release-збірці; усі працюють.

### Формат файлу

JSON-масив об'єктів з полями `id`, `login`, `email`, `password`, `created_at`:

```json
[
  {
    "id": 1,
    "login": "alice",
    "email": "alice@example.com",
    "password": "pw-alice",
    "created_at": "2026-04-27 20:28:29"
  }
]
```

Поле `password` зберігається у **відкритому вигляді** — це і є суть
експорту. Якщо вам потрібен бекап БД без розшифровки — просто
скопіюйте `users.db` (див. розділ "Бекап").

### Що робить export

- Розшифровує всі записи поточним master-паролем.
- Серіалізує у pretty-printed JSON.
- На Unix створює файл з правами `0600` (тільки власник). На Windows — стандартний `std::fs::write` (нові файли успадковують ACL батьківської теки; існуючі файли зберігають свій ACL при overwrite).
- Друкує stderr-попередження про plaintext-вміст.

### Що робить import

- Парсить JSON-масив.
- Для кожного запису бере `login`, `email`, `password` (інші поля ігноруються).
- Пропускає записи з порожнім `login` або порожнім `password` (або відсутнім полем `password`) — без помилки, мовчки.
- Шифрує **поточним** master-паролем БД-приймача (не master-паролем БД-джерела).
- Вставляє у БД. По `login` спрацьовує UNIQUE-обмеження.
- За замовчуванням дублікати по `login` пропускаються (повертає лічильник нових). З `--no-skip-duplicates` перший дублікат повертає помилку і зупиняє імпорт (попередні вставки лишаються).

### Round-trip між двома БД з різними master-паролями

```bash
# БД A → JSON → БД B (різні master-паролі)
pwm --db a.db export shared.json     # розшифровує master-A
pwm --db b.db import shared.json     # шифрує master-B
rm shared.json                        # ВАЖЛИВО — plaintext!
```

### Edge-cases (перевірено наживо)

| Сценарій                                | Результат                                             |
|-----------------------------------------|-------------------------------------------------------|
| Експорт з порожньої БД                  | `[]` у файлі, exit 0                                  |
| Імпорт `[]`                             | "Імпортовано 0 нових", exit 0                         |
| Імпорт неіснуючого файлу                | `Помилка: reading import file: No such file...`, exit 1 |
| Імпорт зламаного JSON                   | `Помилка: parsing JSON array: ...`, exit 1            |
| Імпорт об'єкта замість масиву           | `Помилка: parsing JSON array: invalid type: map...`, exit 1 |
| Запис з порожнім login                  | Пропускається мовчки                                  |
| Запис з порожнім password               | Пропускається мовчки                                  |
| Запис без поля `password`               | Пропускається мовчки                                  |
| Дублікат login (default)                | Пропускається; лічильник не збільшується              |
| Дублікат login (`--no-skip-duplicates`) | `Помилка: login "X" already exists`, exit 1           |
| Спецсимволи в паролі (Unicode, лапки, бекслеші, пробіли) | Зберігаються побайтово              |
| 1000 записів round-trip                 | Усі 1000 збігаються побайтово                         |

### Безпека експорту

- Експортний файл містить **усі ваші паролі у відкритому вигляді**.
- `0600` на Unix захищає від інших користувачів машини, але:
  - Якщо файл потрапить у git/Dropbox/iCloud — паролі скомпрометовано.
  - Якщо машина бекапиться у незашифроване сховище — теж.
- **Видаляйте експортний файл одразу після використання**: `shred -u backup.json` (Linux) або `rm -P backup.json` (macOS).

## Auto-lock — налаштування і поведінка

```bash
PM_AUTO_LOCK_SECONDS=0 pwm        # вимкнути блокування
PM_AUTO_LOCK_SECONDS=60 pwm       # 60 секунд
PM_AUTO_LOCK_SECONDS=600 pwm      # 10 хвилин
unset PM_AUTO_LOCK_SECONDS; pwm   # 300 секунд за замовчуванням
```

| Значення     | Поведінка                                              |
|--------------|--------------------------------------------------------|
| не задано    | 300 секунд (5 хвилин)                                  |
| `0`          | Auto-lock повністю вимкнено                            |
| додатне ціле | Блокувати після N секунд бездіяльності                 |
| некоректне   | Падає до значення за замовчуванням (300 с)             |

**Що скидає таймер.** Будь-яка успішна дія меню (1–8, 10) скидає
таймер. Пункти 9 (вихід) та 11 (генератор) — **не** скидають.

**Коли таймер перевіряється.** Після того, як ви ввели номер пункту й
натиснули Enter. Якщо ви залишили `pwm` на промпті меню (не натискали
нічого) — ключ лишається в RAM до наступного натискання клавіші. Це
відоме обмеження (M3 в `docs/audit-2026-04.md`); workaround — закривайте
`pwm` коли відходите.

**Що відбувається при блокуванні.** Сесія друкує:
```
Сесію заблоковано через бездіяльність (>300 с). Введіть master password.
```
Дається 5 спроб ввести master-пароль. Після 5 невдалих спроб `pwm` виходить з кодом 1.

## Скрипти й автоматизація

### Pipe master-пароля з файлу

```bash
# ❌ ПОГАНО — пароль у history shell
pwm get github <<< "master"

# ✅ ОК — пароль у файлі-секреті з 0600
chmod 600 ~/.config/pwm-master
cat ~/.config/pwm-master | pwm get github
```

### Pipe з менеджера паролів ОС

```bash
# Linux: secret-tool
secret-tool lookup pwm master | pwm get github

# macOS: keychain
security find-generic-password -a $USER -s pwm-master -w | pwm get github
```

### Скрипт-обгортка для clipboard

```bash
#!/usr/bin/env bash
# pwm-copy: скопіювати пароль у буфер обміну
set -euo pipefail
login="${1:?usage: $0 <login>}"
master_file="${PWM_MASTER_FILE:-$HOME/.config/pwm-master}"

if command -v wl-copy >/dev/null 2>&1; then
  cat "$master_file" | pwm get "$login" | tr -d '\n' | wl-copy
elif command -v pbcopy >/dev/null 2>&1; then
  cat "$master_file" | pwm get "$login" | tr -d '\n' | pbcopy
elif command -v xclip >/dev/null 2>&1; then
  cat "$master_file" | pwm get "$login" | tr -d '\n' | xclip -selection clipboard
else
  echo "no clipboard tool found" >&2
  exit 1
fi
echo "пароль скопійовано в буфер обміну"
```

### Bulk-додавання

```bash
#!/usr/bin/env bash
# Додати кілька акаунтів з CSV (login,email,password):
master_file=~/.config/pwm-master
master=$(cat "$master_file")

while IFS=, read -r login email password; do
  printf '%s\n%s\n' "$master" "$password" \
    | pwm add --login "$login" --email "$email" --stdin
done < accounts.csv
```

### Автоматична ротація master-пароля

```bash
#!/usr/bin/env bash
old_master=$(cat ~/.config/pwm-master)
new_master=$(pwm gen --length 32)

printf '%s\n%s\n%s\n' "$old_master" "$new_master" "$new_master" \
  | pwm change-master

echo -n "$new_master" > ~/.config/pwm-master
chmod 600 ~/.config/pwm-master
```

## Бекап і перенесення на інший комп'ютер

### Спосіб 1 — копіювання `users.db` (рекомендовано)

Зашифрована БД сама по собі є бекапом. На приймачі master-пароль
залишається той самий.

```bash
# Скопіюйте всі три файли (sidecar-и SQLite WAL):
scp users.db users.db-wal users.db-shm new-host:~/passwords/

# Або просто чистий головний файл після `pwm` exit (тоді WAL злитий):
pwm --db users.db <<< $'master\n9\n' >/dev/null   # викликає WAL checkpoint при exit
scp users.db new-host:~/passwords/
```

Перевага: master-пароль не потрібен; БД переноситься у зашифрованому
вигляді.

### Спосіб 2 — JSON-експорт + імпорт (для зміни master-пароля)

```bash
# На джерелі:
pwm --db users.db export portable.json

# Безпечно перенесіть файл на цільову машину
# (наприклад через encrypted USB або `gpg --symmetric portable.json`)

# На приймачі:
pwm --db new.db init           # новий master-пароль може відрізнятися
pwm --db new.db import portable.json
shred -u portable.json         # ОБОВ'ЯЗКОВО — plaintext!
```

Перевага: нагода змінити master-пароль одночасно.
Недолік: файл містить plaintext.

## Зміна master-пароля

### Інтерактивно

```
Виберіть пункт: 10
Поточний master password: ********
Новий master password: ********
Підтвердіть новий master password: ********
Master password змінено. Перешифровано 12 акаунтів під новим ключем.
```

### Через CLI

```bash
pwm change-master
# → старий → новий → підтвердження нового
```

### Особливості

- Атомарно: усе виконується в одній SQLite-транзакції. Якщо щось зламається — БД лишиться у старому консистентному стані.
- Перешифровує **усі** записи новим ключем. Час залежить від кількості записів (на тисячах рядків — все одно секунди, бо Argon2id виконується лише раз для деривації нового ключа, а Fernet-операції на одному рядку — мікросекунди).
- Salt і verifier також оновлюються — стара БД більше неприступна навіть зі старим master-паролем.
- Після зміни — версія KDF мігрує на Argon2id (якщо була legacy PBKDF2).

## Міграція зі старої Python-версії

Rust-версія читає той самий формат БД, що й Python-версія
[`passwordmanagerpy`](https://github.com/zagordenis/passwordmanagerpy).

```bash
# Скопіюйте Python-БД у місце, де запускається Rust:
cp ~/old-python/users.db ./users.db

# Перший запуск:
pwm list
# → попередження: "Увага: ця БД використовує старий KDF (PBKDF2). Змініть master password (пункт 10) щоб мігрувати на Argon2id..."

# Мігруйте на Argon2id (без зміни master-пароля по суті — введіть той самий двічі):
pwm change-master
# → введіть поточний master-пароль
# → введіть той самий пароль як новий (двічі)

# Тепер KDF — Argon2id, попередження більше не з'являтиметься.
```

Деталі — [`MIGRATION.md`](MIGRATION.md).

## Часті проблеми (troubleshooting)

### "DB has no master password yet — run `pwm init` first"

Ви запустили підкоманду на свіжій БД, де ще не задано master-пароль.
Спочатку виконайте `pwm init`.

### "invalid master password"

Невірний master-пароль. У не-інтерактивному режимі видається одразу й
exit code = 1. У інтерактивному меню дається 5 спроб.

### "master password already set"

Ви викликали `pwm init` на БД, де master-пароль уже встановлено. Якщо
треба змінити пароль — використовуйте `pwm change-master`. Якщо треба
почати з нуля — видаліть `users.db` (зашифровані дані будуть втрачені).

### "Помилка: parsing JSON array: ..."

JSON-файл імпорту або зламаний, або не є масивом верхнього рівня.
Перевірте через `python3 -m json.tool < file.json`.

### "Помилка: reading import file: No such file or directory"

Шлях невірний. Перевірте поточну теку (`pwd`) і відносні шляхи.

### "Помилка: login "X" already exists"

Запис з таким `login` вже існує. Використовуйте `pwm update <login>`
(замінити пароль) або `pwm rm <login>` (видалити, потім додати знову),
або при імпорті дайте `--skip-duplicates` (за замовчуванням).

### Auto-lock не спрацьовує під час тривалого простою

Перевірте таймер — поточне значення видається тільки при блокуванні.
Якщо `PM_AUTO_LOCK_SECONDS=0` — він вимкнений. Якщо ви залишили `pwm`
на промпті меню — таймер перевіряється тільки після Enter (відоме
обмеження, M3).

### Помилка `Не TTY` / `EOF` при piping

Master-пароль читається з першого рядка stdin у не-TTY режимі. Якщо
pipe закінчився раніше — буде EOF і exit. Перевірте, що ви передали
достатньо рядків (master + при потребі password окремо).

### `pwm` не знайдено в PATH

Або запускайте по абсолютному шляху (`./target/release/pwm`), або
встановіть глобально: `cargo install --path .` і додайте
`~/.cargo/bin` у `$PATH`.

## Коди виходу

| Код | Коли                                                       |
|-----|------------------------------------------------------------|
| `0` | Команда виконалася успішно                                 |
| `1` | Будь-яка помилка: невірний пароль, не знайдено, IO, parse, тощо |

Stderr містить опис помилки українською (`Помилка: ...`); stdout — лише
очікуваний вивід команди (наприклад, тільки пароль для `pwm get`).
